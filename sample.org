#+TITLE: Oppi — The Chaos Donkey's Guide to Mobile Agent Supervision
#+AUTHOR: Chen
#+STARTUP: nofold
#+OPTIONS: toc:3

* Preface

You built an agent harness. Then you wanted to watch it from your phone.
Then you wanted to *steer* it from your phone. Then you wanted it to ask
you things while you're buying groceries. Then you wanted six of them
running in parallel while you're buying groceries.

This is the story of how a weekend project became a distributed system
and nobody is sorry.

* Part I: Why Supervise Agents from a Phone?

** The Laptop Lid Problem

Agents run on your server. Your server is in a closet. Your laptop is
closed because you walked to the kitchen. The agent has a question. It
waits. You come back twenty minutes later. The context window has moved
on. The agent wrote a haiku about waiting instead.

Mobile supervision solves this: the agent pings your phone, you tap
=Allow=, the agent continues, you go back to making coffee.

** The Grocery Store Principle

The best time for an agent to need approval is when you're in aisle 7
comparing oat milks. This is not a design principle anyone chose. It is
a law of nature, like gravity or the fact that =node_modules= is always
larger than the project.

Oppi leans into this. Push notifications. Haptic taps. Glanceable
session cards. You should be able to supervise a refactoring from the
checkout line.

** Trust but Verify (but Mostly Verify)

Agents are confident. They will tell you they fixed the build while
introducing a retain cycle. They will refactor your networking layer
into something elegant that doesn't compile. They will write beautiful
tests that test nothing.

Your phone becomes the trust boundary. Every tool call, every file
write, every =git commit= — visible, approvable, reversible. The agent
proposes; you dispose (from a park bench, ideally).

* Part II: Architecture of a Supervision System

** The Server: A Mac in a Closet

#+begin_quote
"Any sufficiently advanced server infrastructure is indistinguishable
from a Mac mini someone forgot about." — Not Arthur C. Clarke
#+end_quote

The Oppi server runs as a macOS app. Not a Docker container. Not a
Kubernetes pod. A genuine =.app= bundle sitting in =/Applications/=
like it pays rent. It manages agent sessions as child processes with
crash recovery, health monitoring, and the quiet dignity of a process
that knows it will be killed by =mac-app.sh reinstall= at 2 AM.

*** Why a Mac App?

Three reasons:

1. Apple containers don't support Unix domain sockets (=ENOTSUP= on
   bind mounts). TCP it is.
2. Symlinks break inside Apple containers (=cpSync= copies them as
   symlinks with missing targets). No =node_modules= for you.
3. =launchd= plists are fine until you need to manage server lifecycle
   from a native UI, at which point you realize you just want
   =Process()= and a health check loop.

The Mac app wraps the Node.js server, restarts it on crash, and
provides a menu bar icon that does exactly one useful thing: show
whether the server is alive.

*** The Event Stream

Everything in Oppi flows through Server-Sent Events. The server
maintains one SSE connection per client, multiplexing all session
updates into a single stream. When an agent emits a tool call, the
server broadcasts it. When you approve from your phone, the approval
flows back through WebSocket. When six agents are running
simultaneously, your phone receives a gentle river of structured JSON
that the timeline reducer turns into UI.

The protocol is versioned. The client negotiates. Unknown message types
are logged and skipped. Forward compatibility is not optional when your
server deploys at 2 AM and your phone updates whenever the App Store
feels like it.

** The iOS Client: SwiftUI and Its Discontents

*** Many Small Stores

Oppi uses many small =@Observable= stores. Not one big one. Not a
Redux-style global state. Many tiny stores, each responsible for one
screen's worth of state.

Why? Because SwiftUI re-renders are viral. Touch one property on a
shared store and every view observing that store updates. With 50+
properties across sessions, timelines, file browsers, and settings,
a monolithic store would turn your phone into a space heater.

Each store is =@MainActor=, =@Observable=, and deliberately isolated.
Cross-store communication happens through explicit method calls, not
shared mutable state. This is more code. It is also the only way to
keep a 120Hz ProMotion display from dropping frames while six agents
stream tool output simultaneously.

*** The Timeline Reducer

The timeline is the heart of the chat experience. It takes a stream of
server events — text deltas, tool calls, tool outputs, approvals,
errors — and reduces them into a flat list of timeline entries.

This sounds simple. It is not.

Tool calls arrive before their outputs. Outputs stream character by
character. Some tool calls nest (an agent spawns a child, the child
emits its own tool calls). The server can replay events on reconnect.
The user can scroll while events stream. The keyboard can appear
mid-stream, shifting the scroll position.

The reducer handles all of this. It maintains a map of pending tool
calls, correlates outputs to their calls, merges streaming text into
existing entries, and emits a stable list that =UICollectionView=
can diff efficiently.

It processes roughly 10,000 events per second on an iPhone 16 Pro
without dropping a frame.

*** Syntax Highlighting: The Unexpected Bottleneck

When an agent writes code, it appears in the timeline with syntax
highlighting. This sounds like a solved problem. It is not, on iOS,
in SwiftUI, with streaming content.

The naive approach: parse the code, apply =NSAttributedString=
attributes, hand it to a =UITextView=. This works until the code is
1000 lines long and updating 30 times per second because the agent
is streaming. At that point, attributed string diffing alone costs
10ms per frame, and you're at 100% CPU rendering a Python script.

Oppi's syntax highlighter is a hand-rolled scanner. No tree-sitter.
No regex engine. Character-by-character classification into token
ranges, then batch attribute application. It handles 20+ languages
at under 1ms for typical code blocks. The trick is doing less: don't
re-highlight unchanged lines, don't build an AST, don't care about
semantic correctness. The user is reading agent output on a phone
screen. "Good enough" highlighting at 120fps beats "perfect"
highlighting at 15fps.

** The File Browser: Reading Your Codebase from the Bus

The file browser lets you navigate your workspace from your phone.
Directory listings, fuzzy search, syntax-highlighted source, rendered
markdown, inline images, PDF viewing, video/audio playback.

The file index is fetched once and cached client-side. Fuzzy matching
runs locally on-device with a hand-tuned scoring algorithm that
weights path separators, consecutive matches, and basename hits. Search
results appear as you type, with matched characters highlighted.

For large files, the client shows a size warning before downloading.
On cellular networks, an additional warning appears because your agent
just wrote a 3MB JSON fixture and you're on a metered connection in
a parking garage.

*** Org Mode Support

Because some of us write in org. The renderer parses org syntax
(headings, bold, italic, code, links, lists, tables, drawers, source
blocks, keywords) and converts it to the markdown rendering pipeline.
This gives org files identical typography and styling to markdown
without a separate renderer.

Sections are foldable. =#+STARTUP: overview= collapses to top-level
headings. =#+STARTUP: content= shows all headings with bodies hidden.
Drawers (=:PROPERTIES:=, =:LOGBOOK:=) render as collapsible UI
elements. Source blocks get syntax highlighting. The org experience
on a phone is better than most desktop editors, which is either a
compliment to Oppi or an indictment of desktop editors.

* Part III: The Feedback Loop

** Measure Everything

Oppi instruments aggressively. Time-to-first-token. Markdown streaming
render cost. Timeline layout duration. Cell configuration time. Session
list computation. WebSocket connect latency. Voice input pipeline
stages.

Every metric flows to Grafana. The release preflight dashboard has 39
panels covering 50 iOS metrics. The server health dashboard has 24
panels covering 29 server metrics. If a frame drops, we know why. If
a connection stalls, we know where. If the app launches slowly, we
know which store initialization took too long.

This is not optional. You cannot optimize what you cannot see. And on
a phone, where thermal throttling, memory pressure, and background
task suspension conspire against you, visibility is survival.

** The Experiment Loop

Autoresearch is a pattern Oppi uses internally: define a metric, write
a hypothesis, run an experiment, measure, keep or discard, repeat.

The iOS build takes 45 seconds. The server type-check takes 1.7 seconds
(down from 4.5 after parallelizing). Test suites run in under 30
seconds. This means the feedback loop from "idea" to "measured result"
is under two minutes.

Two minutes is fast enough that you try things. You don't argue about
whether approach A or B is faster — you measure both and keep the
winner. This compounds. A hundred two-minute experiments over a month
produce more optimization than a week of upfront design.

** Error Analysis

Agents make mistakes. Build failures, test failures, type errors, lint
violations. Oppi tracks these across sessions, categorizes them,
identifies patterns. If five agents in a row fail because they forgot
to run =xcodegen generate=, that's not an agent problem — it's a
missing guardrail.

Session error analysis runs periodically. It scans JSONL trace files,
extracts build errors, and reports recurring patterns. The output
feeds back into =AGENTS.md= instructions, which feed into agent
behavior, which reduces error rates, which shows up in the next
analysis. The loop closes.

* Part IV: Concurrency and Chaos

** Six Agents, One Repo

Multiple agents can run against the same workspace simultaneously.
This is powerful and terrifying. Two agents editing the same file is
a merge conflict. Two agents running =xcodebuild= is a SQLite lock.
Two agents running =git checkout= is data loss.

Rules:

- Agents use =sim-pool.sh= for simulator builds (slot-based locking
  with isolated =DerivedData= paths)
- Sub-agents must not run builds, tests, or =xcodegen generate=
- Sub-agents must not run =git checkout=, =git stash=, or modify
  files outside their assigned scope
- The parent agent does integration verification after sub-agents
  finish

These rules exist because they were each violated at least once, with
consequences ranging from "mildly confusing" to "three hours of git
forensics."

** The Stash Mystery

One evening, three sub-agents were dispatched to edit different files.
All three reported success. All three files were empty on disk. The
edits were found later in =git stash=. To this day, nobody knows how
sub-agent edits ended up in the stash. The working theory involves
a race condition between the edit tool's file write and something
in the VS Code file watcher. The practical solution was: write files
with =python3= instead of the edit tool.

Distributed systems are distributed systems, even when the distributed
system is three Claude instances sharing a working directory.

** Swift Concurrency: A Love Letter

Swift 6 strict concurrency checking is enabled. =SWIFT_STRICT_CONCURRENCY: complete=.
All =@Observable= classes are =@MainActor=. There are no force unwraps
in production code. The compiler enforces all of this.

This is wonderful until you need to do anything on a background thread.
=SwiftUI.task= inherits =@MainActor= isolation. If you put heavy work
in a =.task= modifier, congratulations: you just blocked the main
thread. Use =Task.detached=.

=Duration= type gets redacted in =os_log=. Use =Int= milliseconds
instead. =os_log .info= is not persisted in device archives. Use
=.error= for performance diagnostics you need to read later.

=NWPathMonitor= fires once with the current state, then fires again
on changes. If you use =withCheckedContinuation= to get a one-shot
reading, cancel the monitor in the callback or it leaks.

These are the kinds of things you learn at 3 AM when the build finally
succeeds and the runtime immediately does something inexplicable.

* Part V: Rendering

** The Rendering Philosophy

Rendering in Oppi follows a few invariants:

1. *Native over web.* No WebViews for content the OS can render natively.
   =UITextView= for text. =PDFKit= for PDFs. =AVPlayer= for media.
   WebViews are for HTML preview only.
2. *Streaming-first.* Every renderer must handle partial content. The
   agent is still typing. The markdown is half-formed. The code block
   hasn't closed yet. Renderers must not crash, stall, or produce
   garbage on incomplete input.
3. *Dark theme, always.* The UI is dark. Exported content is light.
   These are different themes with different palettes. =RenderTheme=
   has both =.fallback= (dark) and =.light= variants.
4. *Frame budget: 8ms.* At 120Hz, you get 8.3ms per frame. Rendering,
   layout, and display must fit. If they don't, the timeline hitches.
   Timeline hitches are tracked as a metric. They are not acceptable.

** Mermaid and LaTeX

Mermaid diagrams and LaTeX equations render natively via =CGContext=.
No JavaScript. No MathJax. No KaTeX. Custom parsers, custom layout
engines, custom renderers drawing directly to Core Graphics.

This is more work than embedding a WebView. It is also 100x faster,
works offline, respects the app's theme, and doesn't require shipping
a JavaScript runtime for what is fundamentally "draw some boxes and
arrows."

Mermaid supports flowcharts, sequence diagrams, class diagrams, state
diagrams, ER diagrams, Gantt charts, pie charts, and git graphs.
LaTeX handles mathematical notation, symbols, fractions, matrices,
and integrals. Both render at device screen scale for retina-quality
output.

The share sheet exports these as rendered images or PDFs. Tap for smart
default (image). Long-press for format picker. Always light theme.
Always device scale. No chrome, no watermarks, no branding.

** The Markdown Pipeline

Markdown rendering is the most complex subsystem in Oppi. It handles:

- CommonMark parsing with extensions (tables, strikethrough, task lists)
- Syntax-highlighted code blocks in 20+ languages
- Inline images loaded from workspace files
- LaTeX math expressions (inline and display)
- Mermaid diagram blocks
- Streaming content with incremental updates
- Segment-based rendering for efficient UIKit view reuse

The renderer breaks markdown into "segments" — contiguous runs of
inline text, code blocks, tables, images, math, diagrams. Each segment
maps to a reusable UIKit view. When streaming content updates, only
affected segments are re-rendered. Unaffected segments keep their
existing views, avoiding expensive layout passes.

This is what makes it possible to stream a 2000-line markdown document
at 30 updates per second without dropping frames. The alternative —
re-rendering the entire document on every update — would be unusable
after about 200 lines.

* Part VI: Voice and Input

** Voice Dictation

Oppi supports voice input. Tap the microphone, speak, the text appears
in the composer. This uses iOS speech recognition, which is good enough
for English and decent for technical jargon after it's seen enough of
your vocabulary.

The pipeline has three stages:

1. *Prewarm*: The speech recognizer is initialized ahead of time.
   Cold start costs ~200ms. Prewarming on app launch eliminates this
   from the user-visible path.
2. *Recognition*: Real-time transcription with intermediate results.
   The composer updates as you speak, showing tentative text that
   refines over time.
3. *Finalization*: When you stop speaking, the recognizer produces a
   final result with higher confidence. This replaces the tentative
   text.

Voice setup time, first-result latency, and prewarm duration are all
tracked as metrics. Because of course they are.

** The Ask Extension

Sometimes the agent needs your input. Not a yes/no approval — a choice.
"Which approach should I take?" "What should this be named?" "Should I
include tests for edge cases?"

The ask extension renders these as tappable cards on your phone. Each
option has a label and optional description. You can select one, select
multiple, type a custom answer, or ignore the question entirely.

The agent sees your response (or lack thereof) and continues. This
turns what would be a blocking conversation into an asynchronous
interaction. The agent asks at 2:14 PM. You answer at 2:47 PM while
waiting for your burrito. The agent was doing other things in the
meantime because it had five more tasks queued.

* Part VII: Operations

** Deployment

Deployment is deliberately boring:

1. =npm run check= passes (TypeScript, ESLint, Knip, Prettier)
2. =mac-app.sh reinstall= rebuilds the Mac app and restarts the server
3. =install.sh --launch= builds and installs the iOS app on device
4. Push notifications confirm everything is connected

There is no CI/CD. There is no staging environment. There is a Mac
in a closet and an iPhone in a pocket. The test suite runs in 30
seconds. Type-checking runs in 1.7 seconds. If it passes locally,
it ships.

This works because the user count is one. When the user count is one,
"staging" is "running it on your phone for five minutes." When the
user count is one, "rollback" is =git revert= and =reinstall=. When
the user count is one, you optimize for iteration speed, not release
ceremony.

** Monitoring

Grafana dashboards cover everything. Two dashboards, 63 panels total.

The release preflight dashboard shows iOS client health: render times,
network latency, voice pipeline, session list performance, frame
budget compliance. Every metric has a reference SLO. Violations are
flagged.

The server health dashboard shows operations: turn duration, time to
first token, WebSocket connections, event throughput, session lifecycle.
These correlate with the iOS metrics to tell a complete story.

When something is slow, the dashboards show where. When something
broke, the dashboards show when. When an experiment improves a metric,
the dashboards confirm the improvement is real and not a measurement
artifact.

** Sentry

Crash reporting via Sentry. Both iOS and server. Breadcrumbs track
the path to a crash. Release health shows crash-free rates per build.

Debug builds on device use a real Sentry DSN (resolved from
=~/.config/pios/sentry-dsn=). TestFlight builds force =SENTRY_DSN=""=
because the one user already has the debug builds and doesn't need
duplicate crash reports.

* Part VIII: Philosophy

** Embrace the Chaos

Software development is not orderly. Agents introduce more entropy,
not less. The correct response is not to fight the entropy but to
build systems that thrive in it.

Oppi's architecture assumes:
- Agents will make mistakes (every action is supervisable)
- The network will fail (reconnection is automatic, state is replayed)
- The server will crash (the Mac app restarts it)
- The phone will lose focus (push notifications bring you back)
- Multiple agents will conflict (isolation rules prevent data loss)
- Metrics will regress (dashboards catch regressions early)

Designing for chaos means designing for recovery. Every failure mode
has a recovery path. Every error is observable. Every action is
reversible. The system doesn't prevent problems — it makes problems
cheap and recoverable.

** First Principles, Always

Strip away assumptions. Find what's actually true. Build from there.

The assumption was "agents need a web UI." The truth was "agents need
a notification on your phone." The assumption was "markdown needs a
JavaScript renderer." The truth was "=UITextView= with attributed
strings is faster and better." The assumption was "the server needs
Docker." The truth was "Apple containers break symlinks and Unix
sockets, just use a Mac app."

Every clever abstraction started as a direct solution to a specific
problem. The abstractions that survived earned their place. The ones
that didn't were deleted without ceremony.

** The Fastest Feedback Loop Wins

Everything in Oppi is optimized for the speed of the feedback loop.
Parallel type-checking (2.6x faster). Bun for script execution (10ms
vs 465ms). Slot-based simulator pools for parallel builds. Incremental
markdown rendering for streaming content. Push notifications for
agent questions.

The goal is: change to signal in under two minutes. If you can verify
a hypothesis in two minutes, you try more hypotheses. If you try more
hypotheses, you find better solutions. If you find better solutions,
the system improves. The loop is the product.

** Solve It Once

Every problem in Oppi is solved once, then encoded. Build errors
become =AGENTS.md= instructions. Common patterns become skills.
Reference implementations become templates. Metrics become dashboards.

The next agent that runs starts smarter than the last one. The next
session inherits the lessons of every previous session. The system
accumulates knowledge like sediment — each layer making the foundation
more stable.

This is not automation. Automation is doing the same thing repeatedly.
This is *accretion* — each iteration making the next iteration better.
The system doesn't just run; it learns. Not in the machine learning
sense. In the "we wrote it down so we don't forget" sense. Which,
honestly, is the more reliable kind.

* Appendix B: Things That Broke

A non-exhaustive list of things that broke and what we learned:

1. *Sub-agent edits vanished into git stash.* Cause: unknown. Fix:
   write files with =python3= instead of the edit tool when things
   get weird.

2. *=ENOTSUP= on Unix domain sockets in Apple containers.* Cause:
   Apple's sandbox doesn't support UDS on bind mounts. Fix: TCP.

3. *Symlinks break in Apple containers.* Cause: =cpSync= copies
   symlinks as-is, targets don't exist inside the container. Fix:
   don't use Apple containers for Node.js.

4. *=os_log .info= not persisted.* Cause: macOS unified logging only
   persists =.error= and above. Fix: use =.error= for anything you
   need to read later. Accept that your debug logs are ephemeral.

5. *=Duration= redacted in =os_log=.* Cause: Apple privacy policy
   redacts non-scalar types. Fix: convert to =Int= milliseconds.

6. *=SwiftUI .task= blocks main thread.* Cause: =.task= inherits
   =@MainActor= isolation. Fix: =Task.detached= for heavy work.

7. *=XcodeGen generate= overwrites =Info.plist=.* Cause: by design.
   Fix: put keys in =project.yml info.properties=.

8. *Two agents building simultaneously lock SQLite.* Cause: shared
   =DerivedData= path. Fix: =sim-pool.sh= with per-slot isolated
   =DerivedData=.

9. *The file was "truncated" but actually contained the literal text
   =[... truncated ...]=.* Cause: the source file was saved in a
   truncated state. Fix: check the source before blaming the renderer.
   Write your own test files.

10. *Pi RPC doesn't stream =thinking_delta=.* It arrives in
    =message_end= content only. Cause: protocol design. Fix: handle
    it in =message_end= parsing.

* Colophon

This document was written in org mode, rendered by Oppi's org renderer,
viewed on an iPhone in a file browser that streams content from a Mac
in a closet over Tailscale. The renderer converted org to markdown AST,
the markdown pipeline produced attributed strings, UIKit laid them out,
and Core Animation composited them onto a 120Hz ProMotion display.

Total time from file open to full render: roughly 47 milliseconds.

The irony of writing documentation about a system while that system
renders the documentation is not lost on us. It's turtles all the way
down. But at least the turtles are fast.
